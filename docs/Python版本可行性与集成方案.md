# Python 版本可行性与集成方案

## 1. 先说结论

这个项目可以改成 Python 版本，但不建议直接“一比一重写整个项目”作为第一步。

更好的路线通常是：

1. 保留 Go 作为核心算法引擎
2. 把分析能力暴露成稳定接口
3. 再从 Python 调用

原因很简单：

- 当前项目的核心算法已经写得比较完整
- 向听搜索、鸣牌枚举、改良计算、打点和风险模块都不少
- 直接重写到 Python，工作量大，回归验证成本也高

所以如果你的目标是“尽快嵌入到 Python 程序”，优先级最高的方案不是重写，而是封装。

## 2. 为什么当前项目不适合被 Python 直接 import

当前仓库几乎全部是 `package main`，说明它现在是“应用程序形态”，不是“库形态”。

这会带来几个问题：

- 对外没有稳定的 Go API
- 输出主要是终端打印，不是结构化结果
- 实时模式依赖 `server.go`、平台消息和控制台交互
- Python 不能直接把它当普通模块调用

不过也有好消息：

- 真正核心的算法大多已经在 `util` 包里
- `model.PlayerInfo` 这种结构也已经比较接近“库接口”
- 静态分析入口其实很集中，便于抽离

这意味着“改造成可嵌入引擎”是现实可行的。

## 3. 方案一：最推荐，封装为本地 HTTP/JSON 服务

### 3.1 思路

保留 Go 算法，在这个项目里新增一层面向程序调用的 API，例如：

- `POST /api/analyze-hand`
- `POST /api/analyze-meld`

Python 端直接发 JSON 请求，拿结构化 JSON 结果。

### 3.2 优点

- 改动成本最低
- Python 集成最简单
- Go 和 Python 进程解耦，稳定性好
- 后续也方便给桌面端、Web、脚本、服务端复用

### 3.3 需要补的工作

当前已有 `server.go`，但现有 `/analysis` 更偏控制台触发，并不是程序友好的结构化接口。

建议新增：

1. 新的 request/response DTO
2. 把 `analysisHumanTiles` 里的打印逻辑拆成“纯计算 + 输出格式化”
3. 返回 JSON，而不是直接 `fmt.Println`

例如返回：

```json
{
  "shanten": 1,
  "best_discard": "4m",
  "results": [
    {
      "discard": "4m",
      "waits_count": 27,
      "avg_next_waits": 5.8,
      "mixed_waits_score": 14.2,
      "avg_agari_rate": 31.4,
      "dama_point": 3900,
      "riichi_point": 6200,
      "furiten_rate": 0,
      "yaku_types": ["平和", "三色"]
    }
  ]
}
```

### 3.4 Python 调用方式

Python 里可以直接：

- `requests`
- `httpx`
- `aiohttp`

去调用本地服务。

如果你的 Python 程序本身也是服务，这通常是维护成本最低的方案。

## 4. 方案二：导出 Go 动态库给 Python 调用

### 4.1 思路

Go 支持通过 `-buildmode=c-shared` 导出动态库，然后 Python 用：

- `ctypes`
- `cffi`

调用。

### 4.2 优点

- 不需要额外启动 HTTP 服务
- 运行时调用延迟更低
- 可以嵌进纯本地 Python 桌面程序

### 4.3 缺点

- Go 导出给 C 的接口设计比较别扭
- 复杂结构体传递不如 JSON 自然
- Windows 下 DLL、依赖、ABI、内存释放要处理仔细
- 调试成本通常高于 HTTP 方案

### 4.4 适合什么情况

如果你的 Python 程序是本地 GUI、对延迟敏感、又不想带一个本地 HTTP 服务进程，可以考虑。

但从工程性来看，仍然通常不如 HTTP/JSON 方案省心。

## 5. 方案三：把 Go 核心拆成命令行子进程，由 Python 调用

### 5.1 思路

让 Python 通过 `subprocess` 启动 Go 可执行文件，传入参数，读取标准输出。

### 5.2 优点

- 最快落地
- 几乎不用重构 Go 代码

### 5.3 缺点

- 当前输出是面向人类阅读的，不稳定
- 文本解析很脆
- 一旦 CLI 输出格式变了，Python 侧就容易坏

### 5.4 什么时候可用

如果你只是做内部工具、临时脚本、PoC，可以先这样接。

但如果你要长期维护，不建议把“解析控制台文本”作为正式接口。

## 6. 方案四：真正重写成 Python 版本

### 6.1 可行，但工作量不小

如果只做“静态何切分析”，Python 重写是可行的。

但如果你想保留目前项目的大部分能力，重写范围其实很大，包括：

- 34 牌表示与转换
- 向听搜索
- 14 张到 13 张的全量枚举
- 改良计算
- 鸣牌组合枚举与食替过滤
- 打点估算
- 和率估算
- 危险度模型
- 役种判断
- 雀魂/天凤消息接入
- 牌谱/观战缓存逻辑

真正麻烦的不是“能不能写出来”，而是“重写后结果能不能跟现有版本对齐”。

### 6.2 Python 重写的优缺点

优点：

- 能直接无缝嵌入 Python 生态
- 更容易和数据分析、机器学习、Web 后端、桌面应用结合
- 团队如果主要写 Python，会更好维护

缺点：

- 搜索与枚举部分性能大概率下降
- 需要大量测试保证逻辑一致
- 初期会花很多时间补基础设施

### 6.3 什么时候值得重写

只有在下面情况比较明显时，才建议真正重写：

- 你的主系统长期只会维护 Python 技术栈
- 你希望深度改动算法，而不是复用现有行为
- 你愿意投入时间做完整回归测试
- 你不介意先经历一段功能不齐的迁移期

## 7. 如果要做 Python 版本，建议怎么拆

不建议一次性整仓重写，建议按算法核心逐步迁移。

推荐顺序：

1. 先迁移牌表示、字符串转换、`PlayerInfo`
2. 迁移向听计算
3. 迁移 13 张分析
4. 迁移 14 张何切排序
5. 迁移鸣牌分析
6. 迁移打点与和率
7. 最后迁移风险模型和实时事件流

这样做的原因是：

- 你可以每迁一层，就拿 Go 版本做对照测试
- 先把“静态手牌分析”跑通，价值已经很高
- 风险模型和平台接入是外围层，可以后补

## 8. 更现实的折中方案：Go 核心 + Python 外壳

这是我最推荐的方向。

你可以把整个系统分成两层：

### 8.1 Go 负责

- 向听搜索
- 何切排序
- 鸣牌分析
- 打点/和率估算
- 危险度估算

### 8.2 Python 负责

- 业务流程
- UI 或 Web 接口
- 日志、数据库、任务调度
- 数据分析
- 用户系统
- 和你现有 Python 程序的整合

这样做的效果通常最好：

- 性能好的部分继续留在 Go
- 你最熟悉、最方便扩展的部分放在 Python

## 9. 具体建议：怎样把这个仓库改成“可被 Python 使用”

### 9.1 第一步：把算法入口做成库接口

建议新增一个独立包，例如：

- `engine`
- `analyzer`

把这些能力从 `main` 包里抽出来：

- 纯手牌分析
- 鸣牌分析
- 返回结构化结果，而不是打印

例如定义：

- `AnalyzeHand(req AnalyzeHandRequest) (*AnalyzeHandResponse, error)`
- `AnalyzeMeld(req AnalyzeMeldRequest) (*AnalyzeMeldResponse, error)`

### 9.2 第二步：把 CLI 输出层和算法层分离

当前 `analysis.go` 和 `cli.go` 混合了：

- 算法调用
- 结果筛选
- 终端着色输出

建议重构成：

1. 算法层只返回结构体
2. CLI 层再负责把结构体打印成文本
3. HTTP 层把同一份结构体编码成 JSON

这一步做完，Python 接入就非常顺了。

### 9.3 第三步：让 Python 通过稳定接口接入

优先顺序建议：

1. HTTP/JSON
2. 子进程 + JSON stdout
3. DLL/so

如果你没有极端性能要求，第一种通常最省事。

## 10. 我对几条路线的综合建议

### 如果你的目标是“尽快能用”

选：

- Go 保留
- 新增结构化 API
- Python 通过 HTTP 调用

### 如果你的目标是“本地单进程嵌入”

选：

- Go 动态库
- Python `ctypes/cffi`

但前提是你愿意承担更高封装成本。

### 如果你的目标是“未来完全 Python 化”

选：

- 先保留 Go 作为基准实现
- 分阶段迁移静态分析内核
- 用大量测试逐步替换

## 11. 我的判断

从这个仓库当前结构看：

- “改成 Python 版本”是可行的
- “直接全部改写”不是最优先、也不是性价比最高的办法
- “内嵌到 Python 程序里”完全可行，而且推荐优先走“Go 引擎 + Python 调用接口”的路线

如果只选一个最实用方案，我建议：

1. 先把 Go 的分析结果改成结构化返回
2. 暴露本地 HTTP JSON API
3. 由你的 Python 程序调用这个 API

这样能最快把现有项目能力接进你的 Python 程序，同时保留后续继续 Python 化的空间。

## 12. 一句话总结

这个项目最适合的不是“马上重写成 Python”，而是“先把 Go 核心算法封装成稳定接口，再由 Python 调用”；只有当你确定长期要完全转向 Python 维护时，再做分阶段迁移会更稳。
